What is Docker and what is a container?#

A container is a lightweight, stripped-down Virtual Machine (VM). These containers are designed to run software/code in a semi-isolated environment with just their configuration files and dependencies.

Imagine setting up a database that requires Python 2.7 to work, but the current version installed on your system is Python 3.2. You can try installing both versions on the same machine, but it might get messy. Docker is containerization software that helps you create a container containing the database software AND Python 2.7. Now you can run the database on your machine, separate from other applications, without messing up the global install of Python.

You can find a lot of containers to download/experiment with at DockerHub. These containers are explicitly designed to work out of the boxâ€”plug-and-play.

Ok, so what am I supposed to do?#

First things first, understand how to download and run a Docker container. To do this, you need to download three things:

  • docker-cli: a tool to interact with Docker containers
  • docker-compose: a tool to run multiple containers at once using a YAML file
  • Colima: the backend which actually hosts the containers. You can also use docker-desktop, which will also do the same thing.

The details for installing are given in the Quickstart guide section.

Quickstart Guide#

To work on this HW, you need a POSIX-compliant environment on your system. If you are using Linux or macOS, then you are POSIX compliant. Users on Windows need to install WSL2.

1) Install Colima (Mac) / Docker Desktop (Win,Linux)#

Colima is a container runtime that provides a lightweight and efficient way to run Docker containers.

brew install colima

2) Docker Command-Line Tool Installation#

The Docker CLI is the primary tool for interacting with the Docker engine.

macOS:

Using Homebrew to install the Docker CLI on macOS:

brew install docker
brew install docker-compose

Windows (via WSL) and Linux:

Within your WSL Linux distribution, you can install the Docker CLI using the package manager. For Debian-based distributions, the commands are:

sudo apt-get update
sudo apt-get install docker-ce docker-ce-cli docker-compose-plugin

You may also add your user to the docker group to avoid using sudo for every command:

sudo usermod -aG docker $USER

Remember to log out and log back in for this change to take effect.

Understanding the files in the HW#

Once you have installed docker-cli, docker-compose, and colima, you can run HW2 containers.

Open your terminal of choice, navigate to the main folder, and run:

docker build -f Dockerfile.server -t server-a .
  • -f specifies the Dockerfile in use
  • -t names the image

A Dockerfile is a set of instructions on how to build a container. Think of special, almost-English syntax that tells Docker what to fetch and how to build a container. Docker reads the Dockerfile and builds an image, which can be run as a container using colima.

To view the newly generated image, type:

docker ps -a

To start the image, type:

docker run --name server server-a

You will have to create a new Dockerfile for this HW, Dockerfile.sniff, so I recommend getting familiar with the format and the syntax. Google is your friend here.

To get inside the container:

docker exec -it server sh
  • exec allows you to execute shell commands inside the container
  • -it interactive, tty
  • server name of the container instance (notice how the image name is different)
  • sh starts an interactive shell (/bin/bash)

Now inside the container, you can navigate around. You will notice a special folder here named flaskapp. This is the same folder that is given inside the HW2 files.

Check out the Dockerfile.server to see how it was copied inside the image and what it was used for.

When you are building multiple containers, you can use docker-compose.

docker-compose up --build -d
  • up means start.
  • --build means read all the Dockerfiles and create a new image.
  • -d means start all images in detached (background) mode.

When you run docker ps -a after this, you will see that there are two containers launched. Use your understanding from Dockerfile.server to see how docker-compose is used to launch multiple containers.

Goal of the HW: ARP Spoofing#

You have to create a new Dockerfile.sniff. This Dockerfile will launch a new container which connects to the same network as server and client. The client runs a simple script which repeatedly queries the Flask app on server. Your goal is to conduct a bi-directional ARP poisoning attack and capture the proof via a .pcap file. Both the spoofing and the network capture happen on the client container.

To conduct the ARP spoofing attack, you have flexibility. You can handcraft a Python file that does it, but an off-the-shelf tool will also achieve this. You can use tcpdump to capture network traffic. For this, you will have to install tcpdump in the container. Use the Dockerfile as guidance again. If you are more familiar with Wireshark or TShark, that works too. But remember, installing a GUI tool inside a container presents new challenges. You can always capture a file in a container and transfer it to the main (host) device to view it via Wireshark.

You can use Dockerfile.client as a template to generate Dockerfile.sniff. The questions below will help guide you towards the answer.
